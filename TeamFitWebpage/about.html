About Team HyperFit

You may be asking yourself, “So, how does it work?” Well, it turns out Team Team’s HyperFit is incredibly simple to use! You simply place your iPhone into a custom built (bought at Best Buy) sleeve, put it on your arm and fire up the app! If logged into Facebook, you will immediately be greeted with your daily FitPoint goal and current progress as well as daily step count information. Swipe to the right and you can begin your workout!

Here, you can choose from a variety of premade exercise including pushups, jumping jacks, squats, and sit-ups. Additional exercises will be able to be added in a future update as well as a “free exercise” mode, allowing you to do whatever added exercise you want in any order you want, hassle free! After each workout, you can see a simple graph of how your heart rate has changed over the course of your workout, indicated by activity level zones. Back to the main screen, you can swipe left to see how many of each exercise you have done for the day to keep track of your entire workout.

Technical Details

So you may be a developer and what to know how everything works under the hood. We shouldn’t tell you, but here at Team Team, we know we’re better than you in every way (the HyperFit app has been known to increase user confidence).

Heart Rate Monitoring

The heart rate monitoring system uses the camera and flash LED to determine user heart rate. Currently, each workout session is set at 30 seconds. For these 30 seconds, the flash LED is lit and the camera is exposed in 3 second chunks to avoid intensive CPU use of leaving it on throughout the duration of the session. During that time, the camera feed is analyzed for changes in color, corresponding to the pulses of blood flowing through the skin. Additionally, the algorithm is modified to include the user’s age to report the activity range the user is currently in.

The application calculates Maximum Heart Rate from an Age Adjusted formula. From the maximum heart rate we calculate the user's heart rate zones from percentages of their maximum heart rate.

Excercise Recognition

One of the core features of HyperFit is the ability to seamlessly recognize exercises performed by the user, without the need for the user to manually specify what exercise is being performed (contrary to what the current UI suggests) or when it starts or stops. To accomplish this, we broke the task up into two separate parts: detecting the start and end of an exercise, and figuring out what that exercise is. First, to detect the start and stop of an exercise, we used filtered accelerometer data. We poll the accelerometer using CoreMotion at 100Hz to gather as many readings of Cartesian acceleration (x, y, and z) as possible. We then take the vector magnitude of each measurement (sum of the squares) and add it to a ring buffer. On a separate dispatch queue, we empty the contents of the ring buffer and examine only the unread data. Here, we utilize an averaging window over all the vector magnitudes, giving us a very smooth array of the average amount of “stuff” happening over time. We check each element of this array to see if this amount of “stuff” goes over a certain threshold (in an edge detection sense). If goes from under the threshold to over, we start recording all accelerometer data, as an exercise has begun. If it goes from over to under the threshold, we stop gathering accelerometer data as we now have a completed exercise!

To figure out what the exercise actually is, we use either SVM or KNN (adjustable) machine learning algorithms from scikit-learn. We trained the algorithms using a separate, training application where we pick an exercise, perform the exercise, and send the data to the server. Once we feel we have a sufficient number of observations of each exercise sent to the server (when we’re too tired to do another sit-up…), we train a model and the HyperFit app is ready to go. Now, not everyone takes the same amount of time to do a similar exercise. Since we gather measurements at a fixed rate, this means that each exercise detected has a variable amount of measurements. Our machine learning algorithms, however, require a fixed feature size to be able to compare each observation and report a predicted exercise label. To solve this problem, we end up down sampling each detected gesture to a fixed number of measurements. Because each exercise is on the order of hundreds of measurements and we down sample to a feature set of 10 measurements, we did not bother with any interpolation. Instead, for non-integer sampling ratios, our algorithm selects the nearest integer index to any non-integer sample indexes, providing a fairly consistent way of extracting features (measurements) that occurred at similar times between similar exercises. Once down sampled, we send the feature set to the server for prediction, get a response, and provide feedback to the UI. Simple, right?

Server Implementation

We used a Tornado Server hosted on the Microsoft’s cloud platform, Azure. The Tornado implementation is a Python script that receives requests from the app to either add observations to a numbered dataset, train a model using scikit-learn with a KNN or SVM machine learning algorithm attached, or request the currently selected model to predict a label for an observation. The server response to the mobile client depends on the type of request sent. If an observation is sent up to be added to the dataset, the server responds by echoing the observation/label pair back to the client. If a request is sent to train a model for the dataset, the server responds with the re-substitution accuracy for both the KNN and SVM models trained. Finally, if a prediction request is sent, the server responds with the predicted label for the observation sent.

We are also utilizing Microsoft Azure’s Backend as a Service Azure Mobile Services to store user information such as FitPoints and Activities.

Project Constraints

Constraint 1: The app must differentiate between the default exercises (doing nothing, jumping jacks, push-ups, sit-ups, and crunches) with a 90% accuracy rate. We will collect this data from 4 users with at least 5 examples per class for each user.

70% of the time it is 100% accurate! Some exercises had problems as they were easily mixed up. Push-Ups were the biggest problem, as they were easily confused with squats and if the delay between going down and up was too long, it would register several exercises, throwing things off even more. Thresholds need to be adjusted a bit to ensure this doesn’t happen.

Constraint 2: The heart rate will be categorized into five different zones (corresponding to common zones used by fitness applications like the Garmin forerunner). These zones classifications will be verified to be 95% accurate across five users in the different core workout sessions.

Since the zone system is used with age and gender inputs, it is difficult to tell EXACTLY how accurate it is. However, over the course of a workout session, it is clear the heartrate graph increases.

Constraint 3:The app must correctly display all workout information (exercises done, repetitions, and heart rate) to the user in a clear and visually pleasing way.

The application is gorgeous. Information on each subsection of the workout is displayed immediately afterwards for the user to see rep count and heart rate over time. Additionally, the user can access daily workout information with a swipe to the right on the main screen.
